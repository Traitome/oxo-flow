## 1. Vision & Philosophy

### Core Insight

> **Scientists don't need another pipeline runner. They need an AI partner that understands bioinformatics.**

### Three-Stage AI Companion

```
  Creation            Monitoring          Interpretation
  ────────            ──────────          ──────────────
  "I have an idea"    "It's running"      "I have answers"
       │                   │                    │
  AI + Search        AI watches          AI writes
  generates          predicts failure    scientific
  pipeline           suggests fixes      narrative
       │                   │                    │
  DAG Editor         Auto-retry          Interactive Q&A
  refine visually    from checkpoint     ask about results
```

### Design Principles

| Principle | Meaning |
|-----------|---------|
| **AI proposes, human decides** | AI never executes mutations autonomously |
| **Know what you know** | Every AI output annotated with evidence source and confidence |
| **Degrade gracefully** | 4-level data perception: file system → user description → naming → template |
| **Single source of truth** | Pipeline state syncs across chat, DAG editor, and proposal views |
| **Works everywhere** | Same binary, three modes: personal/team/HPC |

### Trust Boundary (Hard Constraint, unchanged from v0.8)

```
❌ AI 不允许:                 ✅ AI 允许:
   直接修改管线                  提案修改（人确认后调确定性 API）
   直接启动执行                  推荐执行配置（人点 Run）
   直接删除数据                  建议清理（人确认）
   静默操作                      每次干预 = 提案 + 解释 + 可撤销
```

AI service 层**零数据库写权限、零进程管理权限、零文件系统写权限**。只有读 + 建议生成。

### 3.2 Three Editing Modalities

All three modify the same Pipeline State — changes sync instantly across views.

| Modality | How | Example |
|----------|-----|---------|
| **DAG Editor** | Drag nodes, connect edges, click to edit params | Drag qualimap node, connect to STAR output |
| **Chat** | Natural language commands | "Change STAR to 32 threads" → Interpreter Agent parses → applies update |
| **Proposal** | AI suggests alternatives, user picks one | "Alternative: HISAT2 instead of STAR" → one-click swap |

### 3.3 Data Agent — 4-Level Degradation

```
Level 3: File System Access
  → Scan real files, probe content, detect adapters, infer groups
  → 100% accurate, fully personalized
  → Output: "15% Illumina adapter detected, 150bp reads"

Level 2: User Description  
  → User provides metadata in chat: "50bp SE, human, 3 reps"
  → Trusted but annotated: "Based on your description..."
  → Output: "50bp single-end (per user description)"

Level 1: File Naming Convention
  → Can see filenames/paths but not content
  → Best guess, explicitly uncertain
  → Output: "Likely PE from _R1/_R2 naming (confidence: 0.6)"

Level 0: Intent Only
  → Pure knowledge, no data
  → Generic template with defaults
  → Output: "Generic RNA-seq template — adjust params to your data"
```

Every finding annotated: `{field, value, confidence, source, evidence}`

### 3.4 DAG Editor — Mixed Mode

- **Default**: Read-only, view-only, safe
- **Edit Mode**: Click [✏️ Edit] to enter, full drag-and-drop editing
- **Operations**: add_rule, remove_rule, connect_edge, disconnect_edge, update_params, replace_tool
- **Validation**: Every edit triggers automatic DAG validation (no cycles, no orphans)
- **Undo/Redo**: Full operation stack per session
- **State sync**: All edits go through `/api/pipeline/{id}/command` — serialized command queue, optimistic frontend update, authoritative backend validation, SSE broadcast to all views

## 4. Stage B: Execution Monitoring

### 4.1 Core Philosophy

> From "scientist watches logs" to "AI watches the run"

```
Traditional:
  Submit → check later → error → Google → fix → resubmit → repeat

AI Companion:
  Submit → AI monitors → predicts OOM → pauses → suggests fix → 
  user confirms → auto-retry from checkpoint → completes → notifies
```

### 4.3 Alert Levels & AI Intervention

| Level | Trigger | AI Action | Needs Approval |
|-------|---------|-----------|---------------|
| 🟢 info | Step completed, progress update | Send notification | No |
| 🟡 warn | Memory >80%, disk >85%, step slow | Analyze + suggest | No (notify only) |
| 🟠 alert | Predicted OOM, known error pattern | **Auto-pause** + diagnose + fix plan | **Yes** (push notification) |
| 🔴 critical | Actual OOM, disk full, process crash | Save checkpoint + diagnose + fix plan + **one-click retry** | **Yes** (explicit auth) |

### 4.4 Smart Retry Flow

When STAR fails with OOM (exit 137):

```
1. Monitor Agent detects exit 137
2. Auto-saves checkpoint (R1 alignment ✅ done)
3. Diagnostics Engine matches: oom_killed, auto_fixable=true
4. AI presents:

   🤖 "STAR alignment OOM. Detected:
       · R1 alignment ✅ completed 
       · R2 alignment ❌ failed (OOM)
       
       Suggested fix: memory 16GB → 32GB
       Resume from checkpoint (R1 preserved)
       
       [resources]
       memory = "32GB"  ← was 16GB
       
       [🔧 Fix & Retry] [📝 Manual Edit]"

5. User clicks [🔧 Fix & Retry]
6. Config updated → checkpoint loaded → only R2 re-runs
```

## 6. AI Agent Architecture

### 6.1 Agent Inventory

| Agent | Responsibility | Triggers |
|-------|---------------|----------|
| **Orchestrator** | Understand intent, dispatch sub-tasks, assemble results | Every user message |
| **Search Agent** | Web search for best practices, tool recommendations | New tool name, uncertain params |
| **Data Agent** | 4-level data perception, samplesheet parsing, reference detection | User provides data path or description |
| **Tool Expert** | Recommend tools/params, check environment availability | Pipeline generation or editing |
| **Validator Agent** | DAG legality (no cycles, no orphans), parameter validation | Every pipeline mutation |
| **Monitor Agent** | Resource prediction, failure detection, proactive alerting | During active runs |
| **Report Agent** | Scientific narrative, interactive Q&A, literature comparison | After run completion |

### 6.2 Agent Communication Protocol

All agents communicate via a uniform protocol:

```json
{
  "agent_message": {
    "agent": "DataAgent",
    "type": "data_report",
    "confidence": "high",
    "data": {
      "read_type": "paired-end",
      "read_length": 150,
      "sample_groups": { "WT": 3, "KO": 3 }
    },
    "evidence": {
      "read_type": { "source": "file_scan", "path": "/data/*_R1.fastq.gz" },
      "read_length": { "source": "fastq_probe", "first_1000_reads": "mean=150bp" },
      "sample_groups": { "source": "samplesheet.csv", "column": "condition" }
    },
    "unknowns": ["adapter_contamination", "reference_genome"],
    "suggestions": ["Run fastp for adapter detection", "Specify reference genome"]
  }
}
```

Every field carries: `{value, confidence, source, evidence}`

### 6.3 Trust Annotation

All AI outputs are classified:

| Category | Meaning | UI Treatment |
|----------|---------|-------------|
| **Fact** | From data or determined by code | Plain text, link to source |
| **Suggestion** | AI recommendation, human must verify | Italic with "💡" prefix |
| **Unknown** | Insufficient information | Yellow background, "⚠️" prefix |

### 6.4 Search Agent — Quality Control

- Sources scored by reliability: PubMed/bioRxiv > established tools docs > community forums > personal blogs
- Low-quality sources annotated: "⚠️ Source: personal blog — verify independently"
- **Never send user data paths or file contents to external search APIs**

## 7. UI Design System

### 7.1 Design Philosophy

| Principle | Meaning |
|-----------|---------|
| **Minimal** | Every pixel serves a purpose. No decoration. |
| **Professional** | Trustworthy tool for bioinformaticians, not consumer app |
| **Stable & Reliable** | Every action has feedback, is undoable, never loses data |
| **Light Theme** | White dominant, subtle grays, NO dark mode — readable under lab lighting |
| **Comfortable** | Font sizing, line-height, contrast optimized for extended use (≥4 hours) |

## 12. API Design

### 12.1 Chat API

```
POST /api/chat/send
  → { session_id?, message, context?: { data_paths?, samplesheet?, intent? } }
  ← SSE stream:
      event: text       → { chunk: "I recommend..." }
      event: agent      → { agent: "DataAgent", status: "scanning", progress: 0.5 }
      event: action     → { type: "confirm_pipeline", data: { pipeline_id, toml, dag } }
      event: error      → { code, message }
      event: done       → { session_id, pipeline_id }

GET  /api/chat/sessions
  → [{ id, title, updated_at }]

GET  /api/chat/sessions/{id}
  → { messages: [...] }
```

### 12.2 DAG Edit API

```
POST /api/pipeline/{id}/command
  → {
      source: "dag_editor" | "chat" | "proposal",
      operation: "add_rule" | "remove_rule" | "connect" | "disconnect" 
                | "update_params" | "replace_tool" | "reorder",
      payload: { ... },
      undo_token: "..."
    }
  ← { success, pipeline_state, dag_json, validation }

POST /api/pipeline/{id}/undo
  → { undo_token }
  ← { success, pipeline_state, dag_json }

POST /api/pipeline/{id}/redo
  → { redo_token }
  ← { success, pipeline_state, dag_json }
```

### 12.3 Data Perception API

```
POST /api/data/perceive
  → { paths?: [...], samplesheet?: "base64csv", description?: "..." }
  ← { data_level: 0-3, findings: [...], warnings: [...], suggestions: [...] }

GET  /api/data/reference/status
  → { installed: [...], missing: [...], download_commands: [...] }
```

### 12.4 Monitor API

```
GET  /api/runs/{id}/live
  ← SSE stream: node_started, node_completed, node_failed, resource_warning, 
                ai_alert, heartbeat

POST /api/runs/{id}/pause
  → { reason: "user_request" | "ai_recommendation" }

POST /api/runs/{id}/resume
  → { from_rule?, memory_adjust?, thread_adjust? }

GET  /api/runs/{id}/ai-status
  → { overall, alerts: [...], resource_forecast, estimated_completion }
```

### 12.5 Report API

```
GET  /api/runs/{id}/report
  → { qc_summary, key_findings, narrative_md, caveats, 
      suggested_next, file_tree, charts }

POST /api/runs/{id}/report/ask
  → { question }
  ← SSE: AI answers step-by-step with data lookups and chart references

GET  /api/runs/{id}/report/export
  → ?format=pdf|html|docx
  ← file download

POST /api/runs/{id}/report/visualize
  → { type: "volcano"|"heatmap"|"coverage"|"pathway", params? }
  ← { plot_json }
```

### 12.6 AI Config API

```
GET  /api/ai/config/user     # Get current user's effective AI config
PUT  /api/ai/config/user     # Update user AI config

GET  /api/ai/config/server   # Get server-wide AI config (admin only)

GET  /api/ai/config/effective # Resolved config per priority rules
```

## Appendix A: Key Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Agent coordination complexity | HIGH | Uniform AgentMessage protocol; start with 2 agents, add incrementally |
| AI API cost at scale | MEDIUM | Response caching, token usage visibility, local Ollama fallback |
| DAG editor conflict resolution | MEDIUM | Serialized command queue, optimistic update, backend authority |
| Search quality poisoning | MEDIUM | Source reliability scoring, never auto-apply search results |
| Data privacy in search | HIGH | Search Agent only sends tool names/params, never user data paths |
| Cross-platform rendering | LOW | System fonts only, sRGB colors, fluid layouts |
